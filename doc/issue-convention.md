# Issue 作成・運用規約

keruta プロジェクトにおける GitHub Issue の作成・運用に関する規約を定める。

## 1. 概要

**すべての作業はIssueに作業計画を記載してから着手すること。**

Issue は作業の目的・計画・進捗を共有するための唯一の手段である。
Issueなしの作業開始は禁止する。

## 2. Issue 作成のタイミング

以下の作業を開始する前に、必ずIssueを作成すること:

| 作業種別 | 説明 |
|---------|------|
| 新機能開発 | 新しい機能を追加する |
| バグ修正 | 既存の不具合を修正する |
| リファクタリング | コードの構造を改善する |
| ドキュメント更新 | ドキュメントを追加・更新する |
| CI/CD変更 | ワークフローや設定を変更する |
| 依存関係更新 | ライブラリ・ツールのバージョンを変更する（手動の場合） |

## 3. Issue タイトル規約

タイトルは内容が一目でわかるよう簡潔に記載する。

```
[モジュール名] 変更内容の概要
```

**例**:
```
[ktse] WebSocket切断時のエラーハンドリングを改善する
[ktcl-k8s] K8sジョブの実行ログをストリーミング対応にする
[docs] Issue作成規約を追加する
```

## 4. Issue 本文（作業計画）の必須項目

Issue 本文には以下の項目を必ず記載すること:

```markdown
## 目的
この作業で達成したいことを1-3行で説明する。

## 背景・問題
なぜこの作業が必要なのか、現状の問題点を説明する。

## 作業計画
- [ ] 具体的な作業ステップ1
- [ ] 具体的な作業ステップ2
- [ ] テスト追加
- [ ] ドキュメント更新（必要な場合）

## 影響範囲
- 影響するモジュール・機能
- 後方互換性への影響有無

## 完了条件
この作業が「完了」とみなされる条件を明確に記載する。
```

## 5. ラベル

以下のラベルを適切に付与すること:

| ラベル | 用途 |
|--------|------|
| `bug` | バグ修正 |
| `enhancement` | 新機能・機能拡張 |
| `documentation` | ドキュメント関連 |
| `refactoring` | リファクタリング |
| `ci` | CI/CD関連 |
| `question` | 調査・検討が必要 |

## 6. Issue と作業ブランチの紐付け

- Issue 番号をブランチ名に含めることを推奨する（任意）
- コミットメッセージやPR本文に `#Issue番号` を記載して紐付けること

**例**:
```
feat(ktse): WebSocket切断時のエラーハンドリングを改善 #123
```

## 7. Issue のクローズ

- 対応するPRのマージをもってIssueをクローズする
- PRの説明文に `Closes #Issue番号` を記載すると、マージ時に自動クローズされる
- 作業途中で方針変更・中断になった場合はその理由をコメントに記載してからクローズする

## 8. 禁止事項

以下の行為は禁止する:

- ❌ Issue なしにブランチを作成・作業を開始する
- ❌ 作業計画を記載せずにIssueを作成して着手する
- ❌ 複数の無関係な作業を1つのIssueにまとめる
- ❌ 完了条件を明記しないままIssueを作成する

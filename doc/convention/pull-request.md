# Pull Request (PR) 作成規約

keruta プロジェクトにおける Pull Request の作成・運用に関する規約を定める。

## 1. 概要

Pull Request (PR) は、開発ブランチの変更を `develop` または `main` ブランチに統合するための手段である。
すべての変更は PR を通じてのみ統合され、直接のコミットは禁止する。

## 2. PR 作成の基本ルール

### 2-1. ブランチの選択

| ターゲットブランチ | 用途 | 保護設定 |
|-------------------|------|----------|
| `develop` | 開発環境への統合（デフォルト） | PR経由のみ |
| `main` | 本番環境へのリリース | PR経由のみ、レビュー必須 |

- **通常の開発作業は必ず `develop` ブランチをターゲットにする**（デフォルト）
- 本番リリース時のみ `main` ブランチをターゲットにする
- `gh pr create` を使う場合は必ず `--base develop` を指定すること

### 2-2. 作業ブランチの命名規則

作業ブランチは以下の命名規則に従うこと（詳細は [CONVENTION.md](../CONVENTION.md#1-2-作業ブランチ) を参照）:

```
{prefix}/{description}
```

| プレフィックス | 用途 |
|---------------|------|
| `feature/` | 新機能開発 |
| `fix/` | バグ修正 |
| `fixes/` | 複数修正を含むブランチ |
| `doc/`, `docs/` | ドキュメント追加・更新 |
| `refactor/` | リファクタリング |
| `changes/` | 既存機能の変更 |
| `renovate/` | Renovateによる自動更新 |

### 2-3. PR の粒度

- **1 PR = 1 機能（または 1 修正）** を基本とする
- 大きすぎる変更は複数の PR に分割する
- 無関係な変更を1つの PR に含めない

## 3. PR タイトルと説明

### 3-1. PR タイトル

PR のタイトルは [Conventional Commits](https://www.conventionalcommits.org/) 形式に従うこと:

```
type(scope): 説明
```

| 項目 | 説明 | 例 |
|------|------|-----|
| `type` | 変更の種類（feat, fix, docs, refactor, ci, chore, test, revert） | `feat` |
| `scope` | 対象モジュール名（任意） | `(kicl-web)`, `(ktse)` |
| `説明` | 何を行ったか（日本語または英語） | `OIDCログイン機能を実装` |

**記述ルール**:
- 説明は現在形で始める（「追加した」→「追加する」）
- 最初の文字は小文字で始める（日本語の場合は問わない）
- 末尾にピリオドは付けない
- 1行目は50文字以内を推奨

**例**:
```
feat(kicl-web): アカウント管理ページを追加
fix(ktse): WebSocket接続が切れる問題を修正
docs: PR作成規約を追加
```

### 3-2. PR 説明（本文）

PR テンプレート（`.github/pull_request_template.md`）に従って記入する。

最低限、以下の項目を含めること:

```markdown
## 概要
変更の目的を1-2行で簡潔に説明。

## 主な変更点
- 変更点1
- 変更点2

## 影響範囲
- 影響を受けるモジュール・機能
- 後方互換性のある/なし

## 関連
- 関連Issue番号
- 関連PR番号
```

## 4. PR 作成前のチェック

PR を作成する前に、以下の項目を確認すること:

### 4-1. コード品質

- [ ] コードスタイルが規約に従っている (`./gradlew ktlintCheck`)
- [ ] 新機能には適切なテストを追加済み
- [ ] バグ修正には再現テストを追加済み
- [ ] 全テストが通過している (`./gradlew test`)
- [ ] 不要なコンソールログやデバッグコードを削除済み

### 4-2. コミット

- [ ] コミットメッセージが Conventional Commits に従っている
- [ ] 論理的な単位でコミットされている
- [ ] 秘密情報（認証情報、APIキーなど）が含まれていない

### 4-3. CI/CD

- [ ] CI が全て通過している（GitHub Actions のステータスを確認）
- [ ] 必要に応じてドキュメントを更新済み

## 5. レビュープロセス

### 5-1. レビュアーの指定

- 最低1名のレビュアーを指定すること
- 変更の影響範囲に詳しいメンバーを指定すること
- レビュアーは変更内容を確認し、承認または指摘を行う

### 5-2. レビュー基準

レビュアーは以下の観点でレビューを行う:

| 観点 | チェック項目 |
|------|-------------|
| 機能 | 要求通りの動作をしているか |
| コード品質 | 規約に従っているか、可読性は高いか |
| テスト | 適切なテストが書かれているか |
| セキュリティ | 脆弱性がないか、秘密情報が含まれていないか |
| パフォーマンス | 性能への影響はないか |
| ドキュメント | 必要なドキュメントが更新されているか |

### 5-3. レビュー指摘への対応

- レビュー指摘は修正コミットで対応する
- 修正内容は明確にコミットメッセージに記載する
- 必要に応じて force push を行ってもよい（ただし、他者がレビュー中の場合は注意）
- すべての指摘に対応後、レビュアーに再レビューを依頼する

## 6. マージ

### 6-1. マージ条件

- 最低1名のレビュー承認を得ている
- CI が全て通過している
- コンフリクトが解消されている

### 6-2. マージ方法

- **Squash Merge** を使用する
- マージ時のコミットメッセージは PR タイトルを使用する
- マージコミットのメッセージは Conventional Commits 形式に従う

### 6-3. マージ後の作業

- 作業ブランチは削除する
- 関連する Issue があれば、クローズする
- 必要に応じて `develop` から派生した他のブランチに変更を反映する

## 7. 特殊なケース

### 7-1. ドラフト PR

- 作業途中でフィードバックが欲しい場合は「ドラフト PR」として作成する
- ドラフト PR はレビュー対象外とする
- 準備ができたら「Ready for review」に変更する

### 7-2. WIP (Work In Progress)

- 大きな変更や実験的な機能は、WIP ラベルを付けて PR を作成してもよい
- WIP PR ではレビューは行わないが、議論の場として使用できる

### 7-3. 緊急修正（ホットフィックス）

- 本番環境の重大なバグ修正は `main` ブランチに対して直接 PR を作成してもよい
- ただし、速やかに `develop` ブランチにも反映すること

### 7-4. Renovate による自動更新

- 依存関係の更新 PR は `renovate/` プレフィックスを使用
- CI が通過していれば自動マージされる（`renovate-merge.yml` により）
- 自動マージに失敗する場合は手動でレビューを行う

## 8. ラベルとプロジェクト

### 8-1. ラベル

必要に応じて以下のラベルを付与する:

| ラベル | 用途 | 付与タイミング |
|--------|------|--------------|
| `bug` | バグ修正 | 任意 |
| `enhancement` | 新機能・機能拡張 | 任意 |
| `documentation` | ドキュメント関連 | **`doc/` 配下の変更を含む PR には必須** |
| `refactoring` | リファクタリング | 任意 |
| `breaking-change` | 後方互換性のない変更 | 任意 |
| `dependencies` | 依存関係の更新 | 任意 |

**ドキュメントラベルの付与ルール:**

`doc/` 配下のファイルを変更・追加・削除する PR には、必ず `documentation` ラベルを付与すること。

```bash
# PR作成時にラベルを付与する例
gh pr create --base develop \
  --label "documentation" \
  --title "docs: ..." \
  --body "..."

# 既存PRへのラベル付与
gh pr edit <PR番号> --add-label "documentation"
```

### 8-2. プロジェクト

- 関連する GitHub Project があれば、PR を追加する
- プロジェクトのステータスを適切に更新する

## 9. 禁止事項

以下の行為は禁止する:

- ❌ 直接 `develop` や `main` ブランチにコミットする
- ❌ テストが失敗したままマージする
- ❌ 秘密情報（認証情報、APIキー等）を含めたままマージする
- ❌ レビューなしで `main` ブランチにマージする（緊急時を除く）
- ❌ 大きすぎる PR を作成する（レビュアーの負担を増やす）
